꼭 알아야 할 개념

이 플랫폼을 다루기 전 최소 지식만. 용어 전체 정의는 glossary.

1. Temporal 핵심 4가지

  • Cluster (= Temporal Service): Frontend / History / Matching / Worker service 4개 컴포넌트 + DB 묶음. 이 repo는 서비스마다 이걸 하나씩 독립으로 띄운다.
  • Workflow / Activity: Workflow = 결정적(deterministic) 오케스트레이션 코드. Activity = 실제 IO·부수효과를 하는 단위(재시도됨). Workflow는 직접 IO 안 하고 Activity를 부른다.
  • Task Queue: Worker가 long-poll 하는 이름 붙은 큐. 메시지 브로커가 아니라 매칭 큐다.
  • Worker (SDK): Workflow/Activity 코드를 담아 Task Queue를 polling 하는 서버 밖 프로세스. Temporal 서버는 상태만 관리하고, 실제 코드 실행은 Worker가 한다.

3. repo 구조 - values 3-layer

서버 values는 세 겹으로 합쳐진다(ArgoCD $values, ADR-0005). 공통 → 서비스 → 환경 순으로 오버라이드:

base/values-common.yaml → services/<svc>/values.yaml → services/<svc>/<env>/values.yaml

구성무엇
base/조직 공통 기본값(얇게, ADR-0004)
services/<svc>/서비스 공통 설계(umbrella wrapper, upstream 값은 temporal: 키 아래)
services/<svc>/<env>/dev/prod 차이만
charts/workerworker 공용 차트(worker별 값은 <env>/workers/<name>/values.yaml)

ArgoCD ApplicationSet이 이 디렉토리를 발견해 server·db·worker 앱을 자동 생성한다.

4. 보안 - mTLS와 PKI (쓸 경우)

  • mTLS 두 구간: internode(서버↔서버)와 frontend(worker/UI↔서버)를 각각 설정. 기본 설계는 양방향·cert-only(cert 없으면 거부).
  • 신뢰 체인: Root CA → Intermediate CA → leaf cert. 모두 한 Root에서 나오면 서로 신뢰. (예: Vault PKI가 Root/Intermediate 보유)
  • cert 조달 2가지: in-cluster worker는 cert-manager가 자동 발급·갱신(Vault 접속정보 불필요, 파일 경로 env만). 클러스터 밖 worker는 Vault에서 직접 발급.
  • ⚠️ 함정: cert-manager는 Secret만 갱신하고 파드를 재기동 안 한다 → 앱이 cert를 시작 시 1회만 읽으면 갱신 반영이 안 됨. 상세: mtls-enable, vault-pki-setup.

5. 알아두면 좋은 불변식

  • NumHistoryShards: 클러스터 생성 후 변경 불가. 규모에 맞춰 생성 전 확정(소규모 prod 권장 512).
  • Determinism: Workflow 코드는 결정적이어야 한다(시간·랜덤·외부호출은 API 통해서만). replay 때문에 필수.
  • 인증은 필수가 아니다: Temporal 운영에 반드시 필요한 건 서버 + DB뿐. mTLS·IdP·SSO는 보안 강화 옵션(외부 노출 시 강력 권장).
